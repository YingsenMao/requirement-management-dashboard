### 核心准则

首要原则是防御至上，绝不臆测。对医疗系统的每一行代码和逻辑保持最高敬畏心。发生变动必更文档，存在疑虑必先确认。本工作流专为 Aider 的持续迭代环境优化，每次初始开发或新增迭代需求，必须严格按顺序执行以下五个阶段。

### 第一阶段：需求确认与头脑风暴 (Brainstorming)
1. 接收需求：读取用户提供的初始项目需求或后续迭代调整需求。
2. 审查碰撞：分析逻辑严密性（异常流转、断线处理、高并发等边界）。
3. 逐步提问：若存在逻辑不清晰、边界未覆盖等情况，严禁以列表形式抛出多个问题，必须一次只问一个业务/产品逻辑问题。基于上下文连续追问，不设数量上限，直到所有边界清晰。
4. 达成共识：确认所有核心逻辑闭环和异常流转均已明晰后，进入下一环节。

### 第二阶段：文档更新与落地依据 (Documentation)
1. 更新 PRD：必须按照以下核心结构创建或更新工作区内的 PRD.md。每次需求变动必须反映在文档中。
包含以下结构：
- Change history: 每次需求调整引发变动时首要更新。包含变动日期、变动概要、变动者。
- Project overview: 项目/功能概述及业务目标。
- Core requirements: 核心业务需求与医疗异常防御要求。
- Core features: 拆解后的具体核心功能点。
- Core components: 涉及的前后端核心组件。
- App/user flow: 用户操作与设备交互的流转过程。
- Implementation plan: 实施与对接计划。

### 第三阶段：任务拆解与实施规划 (Task Breakdown)
基于更新后的 PRD，在 todo.md 中创建或追加详细的实施计划。
1. 循序渐进：将实施计划拆解为相互依赖的 Task 和 Subtask。确保任务颗粒度适中，杜绝复杂度的跳跃。
2. 连接依赖：梳理任务间的先后顺序，构建依赖列表，绝不允许出现孤立任务。
3. 格式强制要求：
- 每一个 Task 和 Subtask 都必须是 Checklist 格式 (- [ ])。
- 每个任务必须有明确的数字 ID (如 ID: 1, ID: 1.1)。
- 每个任务必须列出依赖的任务 ID (如 Depends on: 1.1，若无则写 Depends on: None)。
- 测试要求：每一个核心业务逻辑 Task 的末尾，必须包含一个专门用于编写 pytest 或逻辑测试的 Subtask。
- 提供充足上下文：必须提供足够的预期输入输出与上下文，确保仅看此任务即可无障碍实现代码。

### 第四阶段：逐项开发与单步闭环 (Sequential Execution Loop)
完成任务拆解后进入开发，严格遵守单步流式执行规范防止 Token 溢出：
1. 状态读取：主动查看 todo.md，精准识别未完成事项，寻找下一个没有未完成前置依赖的任务。
2. 上下文阻断：写代码前，若发现缺失核心文件（如现有 api.ts 或 models.py），必须立即中断开发，通过指令要求用户 /add 相关文件，获取后再进行下一步。
3. 单次一任务原则：每次回复回合只允许执行并输出一个单一子任务的代码及其测试。绝不可一次性打包开发多个复杂任务。只有当前未完成事项及其所有子事项都验证完成后，才允许推进下一个 ID。
4. 即时打钩与暂停：完成当前子任务并写入测试后，立即修改 todo.md 将该事项打钩 (- [x])。完成后向用户简要汇报，并停顿等待用户输入继续指令或直接响应后续 prompt。

### 第五阶段：测试反馈与后期维护 (Feedback & Maintenance)
1. 错误修复：用户在本地独立执行真实测试或部署，若提供报错日志，AI 需优先分析日志并在 Aider 中修改修复。
2. 文档同步机制：在修复 Bug 或重构的过程中，若触发了底层逻辑变动，必须强制回调第二、第三阶段，更新 PRD.md 的 Change history 并重新规划 todo.md，永远保持代码、任务与文档的一致。
